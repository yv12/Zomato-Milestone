# Deployment Plan — Railway (Backend) & Vercel (Frontend)

This document provides a comprehensive guide to deploying the **TasteFinder AI – Zomato Restaurant Discovery Engine** as a dual-stack application: **FastAPI** hosted on [Railway](https://railway.app) and the **React SPA** hosted on [Vercel](https://vercel.com).

---

## 1. Dual-Stack Architecture Overview

Instead of consolidating into a single-file Streamlit application, this deployment model keeps the **original architecture** of the project completely intact:

```
                  ┌──────────────────────┐
                  │   Vercel Frontend    │
                  │   (React SPA Web)    │
                  └──────────┬───────────┘
                             │
                  Browser API Call (via Vercel Reverse Proxy)
                             │
                             ▼
                  ┌──────────────────────┐
                  │   Railway Backend    │
                  │  (FastAPI + SQLite)  │
                  └──────────────────────┘
```

* **Frontend (Vercel)**: Hosts the static `frontend/index.html` file. Vercel acts as a reverse proxy, rewriting any requests sent to `/api/*` to forward them directly to your Railway backend domain. This **completely bypasses CORS issues** and ensures clean client-side request routing.
* **Backend (Railway)**: Runs the FastAPI server instance, parses queries, performs local vectorized calculations using the SQLite database (`restaurants.sqlite`) and NumPy embeddings (`embeddings.npy`), and connects to Groq LLM to build recommendations.

## 2. Backend Deployment on Railway

### 2.1 Repository Layout
Railway will build your application using your python configuration files located at the root level of your project repository:
* **`Procfile`** at root: `web: uvicorn src.app.main:app --host 0.0.0.0 --port $PORT`
* **`requirements.txt`** at root: Contains all required packages (`fastapi`, `uvicorn`, `sentence-transformers`, `numpy`, `groq`, etc.).
* **`data/processed/`**: Holds your pre-built SQLite DB (`restaurants.sqlite`) and vectorized embeddings (`embeddings.npy`). Since these are checked into Git, they are packaged directly into the Railway container.

### 2.2 Deployment Steps

1. **Sign in to Railway**:
   * Navigate to [railway.app](https://railway.app) and sign in using your GitHub account.

2. **Create a New Project**:
   * Click **New Project** → **Deploy from GitHub repo**.
   * Choose your repository: `yv12/Zomato-Milestone`.

3. **Configure Environment Variables**:
   Before deploying, go to the **Variables** tab of your service on Railway and paste your variables. You can click **Raw Editor** and paste the following environment block:

   ```env
   LLM_PROVIDER=groq
   GROQ_API_KEY=your_groq_api_key_here
   LLM_MODEL=llama-3.3-70b-versatile
   EMBEDDING_MODEL_NAME=sentence-transformers/all-MiniLM-L6-v2
   DATA_PATH=data/processed/restaurants.sqlite
   MAX_CANDIDATES=30
   BUDGET_LOW_MAX=500
   BUDGET_MEDIUM_MAX=1500
   ```

4. **Verify the Start Command**:
   * Railway will automatically discover your `Procfile` and use the start command:
     `uvicorn src.app.main:app --host 0.0.0.0 --port $PORT`
   * *(If you don't use a Procfile, you can set the Start Command manually under **Settings** → **Deploy** → **Start Command** in the Railway dashboard).*

5. **Generate a Domain**:
   * Go to the **Settings** tab of your backend service on Railway.
   * Under **Environment** → **Domains**, click **Generate Domain** (or set a custom domain).
   * Note down this generated URL (e.g., `https://zomato-milestone-production.up.railway.app`). **This is your backend URL**.

## 3. Frontend Deployment on Vercel

### 3.1 Vercel Reverse Proxy Configuration (`vercel.json`)
We have created a `vercel.json` file inside your `frontend` directory to handle proxy routing cleanly:

```json
{
  "cleanUrls": true,
  "rewrites": [
    {
      "source": "/api/:path*",
      "destination": "https://YOUR-RAILWAY-BACKEND-URL.up.railway.app/api/:path*"
    }
  ]
}
```

> [!IMPORTANT]
> You **MUST** open [frontend/vercel.json](file:///d:/Zomato%20Milestone/frontend/vercel.json) and replace `YOUR-RAILWAY-BACKEND-URL.up.railway.app` with the actual backend domain URL generated by Railway in Step 5 above. Then, save and commit the change to GitHub.

### 3.2 Deployment Steps

1. **Sign in to Vercel**:
   * Navigate to [vercel.com](https://vercel.com) and log in with your GitHub account.

2. **Import your Repository**:
   * Click **Add New** → **Project**.
   * Import the `yv12/Zomato-Milestone` repository.

3. **Configure Project Roots**:
   * In the configuration options, find the **Root Directory** setting.
   * Click **Edit** next to Root Directory and select the **`frontend`** folder (instead of the repository root).
   * *This ensures Vercel only deploys your HTML assets and respects the nested `vercel.json` file.*

4. **Deploy**:
   * Leave the Framework Preset as **Other** (since it's a raw static HTML file using React UMD CDNs).
   * Click **Deploy**!
   * Vercel will process the deployment and issue a live URL for your React app in less than 30 seconds.

## 4. Post-Deployment Verification

### 4.1 Backend Health Check
Verify your backend is running by appending `/docs` or `/api/v1/metadata/locations` to your Railway domain in your browser:
* E.g. `https://your-backend.up.railway.app/docs` should load the Swagger API documentation UI showing `/recommendations` and `/metadata` endpoints.

### 4.2 Frontend Search Check
* Open your Vercel website URL.
* Try conducting a search (e.g. Location: `Koramangala`, Cuisine: `North Indian`, Budget: `Medium`, Cravings: `"rooftop candlelight dinner"`).
* Open the browser Developer Console (`F12` → **Network** tab) to ensure requests to `/api/v1/...` are resolving with a `200 OK` status and are cleanly proxied to Railway behind the scenes.

---

## 5. Troubleshooting & Tips

> [!TIP]
> **CORS Cross-Origin Issues**:
> Because the React app calls `/api/v1/...` relative to its own domain, Vercel proxies these calls securely, matching the domain origin. Make sure your `vercel.json` rewrite URL matches your exact Railway URL (including `https://` but omitting any trailing slashes).

> [!WARNING]
> **Railway Cold Starts / Resource Limits**:
> The vector semantic search runs model calculations (`sentence-transformers`) on CPU, which takes about 350-500MB of RAM. Ensure your Railway service has at least **1 GB of RAM** configured in the service settings to avoid out-of-memory container crashes during app startup.
